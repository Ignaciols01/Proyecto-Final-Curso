RosterApp

Memoria del Proyecto

Documentación cronológica de las fases de análisis, diseño, desarrollo y estado actual del aplicativo web para la gestión de cuadrantes.

Desarrollador: Ignacio Leo Sánchez

Centro: IES Albarregas, Mérida

Módulo: Proyecto de Desarrollo de Aplicaciones Web

Introducción

El presente documento detalla de manera exhaustiva el ciclo de vida del desarrollo de RosterApp. Para garantizar una ejecución metódica y profesional, el proyecto se ha dividido en 15 hitos o fases secuenciales, abarcando desde la concepción inicial de la idea hasta su estado actual. A continuación, se expone el trabajo realizado en cada una de estas etapas.

01 Presentación de Asignatura y Proyecto

En esta fase inicial se definieron los objetivos generales exigidos por el módulo. Se identificó una problemática real en el sector minorista: la ineficiente gestión manual de los turnos de trabajo. Como solución, se propuso el desarrollo de "RosterApp", una Single Page Application (SPA) capaz de centralizar los horarios, separar la lógica mediante roles (Gerente/Empleado) y eliminar la dependencia de grupos de mensajería o tablones físicos.

02 Creación de Imagen Corporativa de la Empresa

Para dotar al software de un aspecto profesional y comercial, se elaboró un manual de identidad visual. Se diseñó un logotipo vectorial representativo y se estableció la tipografía "Montserrat" por su alta legibilidad en interfaces digitales. Asimismo, se definió una paleta de colores planos y funcionales: Azul Primario (#0062cc) para la marca y acciones principales, Azul Secundario (#138496) para acentos, Verde Éxito (#218838) para confirmaciones y turnos de tarde, y Naranja Aviso (#e0a800) para alertas y turnos especiales.

03 Elaboración de Contrato y Recogida de Necesidades

Para simular un entorno laboral real, se redactó un Contrato de Prestación de Servicios Informáticos con la empresa ficticia "Maderas Termita Veloz, S.L.", ubicada en Cádiz. En este documento se estipularon las cláusulas legales, obligaciones de confidencialidad, y un plazo de ejecución estricto de 5 meses. Durante esta fase se recogieron las necesidades críticas del cliente: un sistema de fichajes, control horario de jornadas de 8 horas, diferenciación de perfiles (Trabajador, Administrador) y generación de informes.

04 Definición de Requisitos Funcionales y No Funcionales

Con el contrato firmado, se tradujeron las necesidades del cliente a requerimientos técnicos.

  • Funcionales: Autenticación segura, CRUD completo de empleados por parte del gerente, creación dinámica de turnos y asignación múltiple de trabajadores.
  • No Funcionales: Arquitectura SPA para rendimiento óptimo sin recargas, diseño 100% Mobile First, y seguridad estricta para garantizar que un empleado solo vea sus propios datos.

05 Desarrollo de las Interfaces Gráficas

Utilizando la guía de estilos previamente definida, se procedió a la maquetación y diseño UI/UX (User Interface / User Experience). Se crearon las pantallas clave de la aplicación: el panel de inicio de sesión dividido, el Dashboard principal del gerente con el calendario semanal, la tabla de gestión de personal, las ventanas modales flotantes para la asignación de turnos, y la vista simplificada en formato lista para la consulta de los empleados. Todo ello programado bajo directrices responsivas.

06 Desarrollo de la Estructura de la Base de Datos

Se configuró el entorno de backend utilizando Supabase (basado en PostgreSQL). Se instanciaron las tablas necesarias para almacenar de forma persistente la información generada por la interfaz. Esto incluyó la configuración de columnas, tipos de datos (UUIDs, VARCHAR, TIMESTAMPS) y la implementación inicial de las reglas de seguridad a nivel de fila (Row Level Security).

07 Definición del Modelo Relacional de la Base de Datos (MER)

Para garantizar la integridad y normalización de los datos (evitando violar la Primera Forma Normal), se elaboró y justificó el MER. La base se estructuró en tres entidades clave: Usuarios (que engloba a administradores y empleados mediante un campo 'rol'), Turnos (bloques de tiempo abstractos) y, para resolver la relación de cardinalidad Muchos a Muchos (N:M), se creó la tabla intermedia Asignaciones con claves foráneas. Esto permite que varios empleados compartan un mismo turno sin duplicar información.

08 Presentación a la Empresa de Interfaces y Base de Datos

Se realizó un hito de control y validación con el "cliente". En esta fase se expusieron los mockups de las interfaces gráficas y la lógica del modelo de datos para demostrar cómo se cumplían los requerimientos del contrato. Tras asegurar que la privacidad de los empleados estaba garantizada mediante la tabla de asignaciones, se aprobó el avance a la fase de programación del código fuente.

09 Elección de Tecnologías a Utilizar

Se cerró el stack tecnológico definitivo (Pila de desarrollo). Para el entorno operativo se utilizó Windows 11. El desarrollo del cliente (Frontend) se llevó a cabo con JavaScript y la librería React, utilizando Tailwind CSS v4 y HTML5 para los estilos y la estructura. Todo el backend y la base de datos se delegaron a Supabase (PostgreSQL).

10 Estructuración Inicial de Documentación

Con el desarrollo técnico en marcha, se procedió a organizar el repositorio del proyecto. Se creó un árbol de directorios separando el código fuente (carpeta src/) de la documentación obligatoria (carpeta docs/). Se redactó el archivo README principal para presentar el repositorio en plataformas como GitHub/GitLab al tutor asignado.

11 Definición de Puntos de los Manuales de Usuario y Técnico

Se elaboraron los esquemas y los índices de los entregables finales. Para el Manual Técnico se decidió incluir la arquitectura, el MER y la justificación de los requisitos. Para el Manual de Usuario, se estructuró un flujo lógico: acceso al sistema, explicación exhaustiva de las herramientas del perfil Administrador (calendario y personal), y explicación de la vista simplificada del Empleado.

12 Desarrollo Inicial de Manuales de Usuario y Técnico

A partir de los índices generados en la fase anterior, se redactó el contenido en profundidad. Se tomaron capturas de pantalla de las interfaces reales generadas en la Fase 5 y se ensamblaron junto a explicaciones paso a paso. Se aplicó el formato corporativo (Montserrat y colores de marca) a estos documentos para asegurar una presentación impecable.

13 Opciones de Despliegue de Aplicativos

Afrontando la recta final, se investigaron los métodos para sacar la aplicación del entorno de código y ponerla en producción. Se analizaron alternativas modernas de hosting en la nube para aplicaciones SPA (como Vercel o Netlify) frente a opciones de despliegue local tradicionales. Se elaboró el Manual de Despliegue documentando los comandos necesarios (npm install, variables de entorno .env) para compilar el proyecto.

14 Pruebas de Despliegue en Entorno Local: TomCat

Se realizaron pruebas de compilación del código React (generación de la carpeta build/dist) y su posterior integración en servidores. Aunque las SPA modernas suelen servirse con Node, se documentaron y ejecutaron pruebas de viabilidad alojando los archivos estáticos generados dentro de servidores tradicionales como Apache Tomcat, asegurando la correcta resolución de rutas y el correcto funcionamiento de las llamadas al backend de Supabase.

15 Presentación de Asignatura y Proyecto (Estado Actual)

Actualmente, el proyecto RosterApp se encuentra en fase de desarrollo activo y mejora continua. Si bien se han cumplido los hitos principales de diseño, estructuración de la base de datos, implementación de interfaces y documentación base, el aplicativo sigue evolucionando. En esta etapa se están puliendo funcionalidades, optimizando el rendimiento del código y preparando el terreno para futuras actualizaciones antes de su despliegue en producción y entrega definitiva.
